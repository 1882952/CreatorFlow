# 可行性评估：数字人长音频分段批量生成

## 1. 结论

该需求整体可行，建议采用“前端操作台 + 本地编排服务 + ComfyUI + ffmpeg”的组合方案落地。

结论级别：

- 总体可行性：高
- 工程复杂度：中高
- 主要风险：连续性效果波动、时长校准、前后端迁移成本

## 2. 关键技术点评估

| 技术点 | 结论 | 信心 | 说明 | Plan B |
|---|---|---|---|---|
| 音频按停顿点切分 | 可行 | 高 | 可通过 `ffmpeg silencedetect` 检测停顿点 | 无停顿点时强制切分 |
| 单段时长不超过 10 秒 | 可行 | 高 | 切分时强约束最大时长 | 直接按固定窗口切分 |
| 分段串行调用 ComfyUI | 可行 | 高 | 当前已有单任务顺序执行逻辑，迁移到本地编排服务即可 | 保留前端顺序执行作为临时回退 |
| 上一段尾帧传给下一段 | 可行 | 高 | 用 `ffmpeg` 从上一段视频抽帧并重新上传 | 若抽帧失败则直接中止任务 |
| 最终拼接与本地落盘 | 可行 | 高 | 本地 `ffmpeg concat` 天然适合该场景 | 失败时保留分段视频供人工拼接 |
| 自动清理中间文件 | 可行 | 高 | 本地服务可直接操作文件系统 | 失败任务仅做手动清理 |
| 页面刷新后的任务恢复 | 可行 | 高 | 用 SQLite 持久化任务和分段状态 | 最差可通过重新拉取任务列表恢复 |
| 视觉连续性完全无缝 | 有风险 | 中 | 尾帧传递可增强连续性，但不能承诺完全无跳变 | 允许后续加入 overlap 段或交叉淡入策略 |
| 工作流时长与音频精确对齐 | 需要验证 | 中 | 当前工作流内有时长换算和音频裁切逻辑 | 调整 `duration` 传参和 `trim_to_audio` 策略 |

## 3. 约束声明

## 3.1 前提条件

1. 本地 ComfyUI 可正常提供 HTTP 与 WebSocket 服务。
2. 本地机器已安装 `ffmpeg` 和 `ffprobe`，并可从命令行调用。
3. 用户可提供 ComfyUI `input/output` 目录配置，或允许编排服务通过上传 API 与下载 URL 访问。

## 3.2 技术假设

1. 所有分段均使用同一工作流模板和同一编码参数。
2. ComfyUI 单段输出格式足以用于后续 `concat` 拼接。
3. 本地任务规模以单机单用户为主，不需要分布式调度。

## 3.3 已知限制

1. 视觉连续性受模型本身影响，不是纯工程问题。
2. 停顿点检测质量依赖音频内容本身。
3. 当前前端架构为纯前端，迁移到“本地编排服务”会增加改造面。

## 4. 核心风险

## 4.1 连续性风险

风险：

- 片段边界存在人物姿态跳变
- 面部细节在切段处不稳定

应对：

- 默认固定提示词
- 默认固定基础种子
- 尾帧传递作为标准流程
- 为后续版本预留 overlap 片段或边界平滑方案

## 4.2 时长对齐风险

风险：

- 视频长度与分段音频长度轻微不一致
- 拼接后出现嘴型错位

应对：

- 第一轮开发加入“时长校准测试”
- 校验当前工作流中 `duration`、`TrimAudioDuration`、`trim_to_audio` 的真实行为

## 4.3 文件清理风险

风险：

- 清理脚本误删最终结果
- 失败任务被过早清理导致无法排查

应对：

- 清理白名单与黑名单分离
- 失败任务默认不自动清理
- 最终结果单独目录保存

## 4.4 迁移风险

风险：

- 前端从“直接调 ComfyUI”切到“调编排服务”时涉及多文件改造

应对：

- 先以特性开关切入
- 分阶段切换执行入口

## 5. Spike 建议

在正式开发前建议完成以下 3 个 Spike：

### Spike-01 时长校准

目标：

- 验证当前工作流在 4 秒、6 秒、8 秒、10 秒输入下的真实输出时长

输出：

- 一份时长映射表

### Spike-02 尾帧抽取质量

目标：

- 验证 `ffmpeg` 尾帧抽取是否稳定，是否需要取倒数第 2 帧或倒数 0.1 秒位置

输出：

- 尾帧抽取参数建议

### Spike-03 Concat 兼容性

目标：

- 验证当前 ComfyUI 输出的多个视频片段能否直接无转码拼接

输出：

- 直接 `-c copy` 可行或需重编码的结论

## 6. 推荐决策

1. 默认采用本地编排服务，不再以前端 `ffmpeg.wasm` 作为主拼接方案。
2. 默认采用 `ffmpeg` 做停顿点分析、抽帧、拼接和清理。
3. 默认固定基础种子，不把“种子递增”作为首版默认策略。
4. 默认只做 `auto` 与 `single` 两种分段模式。

## 7. 通过门槛

满足以下条件后即可进入实施：

- 已确认编排服务技术栈
- 已确认 `ffmpeg` 依赖策略
- 已确认任务元数据持久化方案
- 已完成 3 个 Spike 的最小验证
